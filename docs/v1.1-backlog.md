# Mars v1.1 Backlog

**Created:** 2026-04-11 (pre-ship prep for Story 9.4)
**Status:** captures everything explicitly deferred out of v1 scope so v1.1 starts the morning after v1 ships with a crisp, force-ranked list.

## How this doc works

- **P0** — ships in v1.1. Real blocker for the next customer.
- **P1** — queued for v1.1 if time allows, otherwise v1.2.
- **P2** — worth capturing, not worth roadmapping.

Every item has a rationale. Padding is the enemy here: if you can't
defend why an item exists, it doesn't belong on the list.

## P1 — fit & finish

### 6. Persisted session registry (replaces v1 default-supervisor URL)
**Rationale:** v1's local-emulation shortcut is
`MARS_DEFAULT_SUPERVISOR_URL` pointing at a single supervisor. The
moment Mars hosts two daemons on different Fly machines, this
breaks and `GET /sessions` returns the wrong list. v1.1 should
replace it with the session registry Epic 5 sketched but
explicitly deferred.
**Scope:** SQLite table on the control plane, fan-out to all
registered supervisors on `GET /sessions`, resolve-by-agent for
`POST /sessions/{id}/input` and `PATCH /agents/{name}/prompt`.

### 7. SSE reconnection with Last-Event-ID
**Rationale:** Native `EventSource` auto-reconnects on network
blip but Mars v1 does NOT honor `Last-Event-ID`, so a reconnect
can drop events. The browser user sees a brief gap in the chat
stream. v1.1 fix: wire the event store's monotonic sequence into
SSE's Last-Event-ID handling so reconnect replays from the last
acknowledged event.

### 8. Permission request round-trip (approve/deny UI)
**Rationale:** v1's PermissionRequest bubble is explicitly
*advisory* — by the time the event reaches the browser, the
PreToolUse hook has already made the decision. `POST
/sessions/{id}/permission-response` returns 501 Not Implemented.
v1.1 wires the approve button to a real callback that unblocks
the daemon.

### 9. Malix design system adoption
**Rationale:** v1 ships ugly Tailwind primitives + one-off
components inlined in each page. Camtom's design system
(`@camtomlabs/malix-design-system`) is the house style; v1.1
adopts it for consistent buttons, forms, modals, and typography.
Not a visual overhaul — just replacing the one-offs.

### 10. Prompt editor UI (Story 6.4 admin-edit completion)
**Rationale:** `PATCH /agents/{name}/prompt` is fully wired on
the backend — the CLI has `mars edit-prompt`, but there's no UI.
v1.1 adds a prompt editor in the chat view's settings drawer so
admins can edit CLAUDE.md without touching a terminal.

### 11. Resumable session flag in the dashboard
**Rationale:** Story 5.4 shipped the backend reconcile logic but
explicitly deferred the `POST /sessions/{id}/resume` endpoint and
the React UI pieces. A "Resume" button on exited-clean sessions
completes the loop.

## P2 — noise register

(These are real observations, but they don't change v1.1 scope.
Capturing them here so future-me doesn't rediscover them.)

- **Single-template limit**. v1 ships one template
  (`tracker-ops-assistant`); adding a second only requires
  dropping another `.yaml` + `.prompt.md` pair into
  `apps/mars-control/templates/`. No code changes needed. Add a
  second template when there's a second customer vertical.
- **Dark mode polish** — Tailwind defaults are fine but some
  contrast ratios on the wizard amber card are slightly off.
- **English toggle** — wizard is Spanish-only in v1. Not a
  blocker; Maat reads Spanish. Add when the second customer
  speaks English.
- **Error retry UX** — the wizard has no retry button on network
  errors. Currently the user closes the modal and reopens it.
- **Zoho MCP** — if the public MCP doesn't exist yet, the
  `tracker-ops-assistant` template will fail the `zoho` MCP
  check at runtime. v1.1 either ships a Zoho MCP (if missing)
  or removes `zoho` from the default template and adds it as
  an optional addon in the wizard.

## Not in backlog — deliberately out of scope

- Multi-tenant isolation (v1 trusts the operator)
- Billing (v1 is free for design partners)
- Public marketplace / template sharing (v2 earliest)
- Teams / invites / RBAC (v2 earliest)
- Customer-writable templates (v2 earliest)
- Admin dashboard for managing users (v2 earliest)

## Ranking rubric

When a new item lands in this backlog, answer three questions:

1. **Does the next customer hit this?** If no → P2.
2. **Does the next customer hit this AND work around it trivially?**
   If yes → P1.
3. **Does the next customer hit this AND give up?** If yes → P0.

If the answer to all three is "I don't know", the item needs more
observation before it can be ranked. Don't rank-by-feeling.
